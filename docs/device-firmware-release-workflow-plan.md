# 设备固件版本发布流程方案

## 1. 方案目标

本方案基于当前已经落地的“设备类别 -> 设备型号 -> 固件版本”管理能力，进一步升级为更贴近实际业务的版本流程管理。

本次重点不是重做整套系统，而是在现有设备固件模块基础上，补齐：

- 开发版本流程
- 发布版本流程
- 发布类型管理
- 发布历史记录
- 发布后禁止删除、只能作废

本方案适用于“按设备型号管理固件版本”，暂不细化到单台设备。

## 2. 核心原则

本次版本管理拆成两条线：

1. 开发版本
2. 发布版本

这样做的原因是：

- 开发阶段关注测试流转
- 发布阶段关注上线、最新、稳定、作废
- 两者业务含义不同，不适合混成一套状态

## 3. 业务定义

### 3.1 开发版本

开发版本表示一个固件包从上传到具备发布条件的过程。

开发版本状态建议为：

- 待测试
- 测试中
- 测试通过
- 测试未通过
- 待发布

含义说明：

- 待测试：固件已上传，但 QA 还没有正式开始测试
- 测试中：QA 正在测试
- 测试通过：测试完成且结果通过
- 测试未通过：测试完成但未通过，需要开发继续修复
- 待发布：版本已经通过测试，等待管理员执行发布

### 3.2 发布版本

发布版本表示一个已经正式进入发布域的固件版本。

发布域不再使用“驳回”概念，只有：

- 已发布
- 已作废

同时保留两个独立标记：

- 最新版本
- 稳定版本

说明：

- 已发布：该版本已经进入发布版本列表
- 已作废：该版本已下线，不再使用
- 最新版本：当前型号下最新对外版本，只能有一个
- 稳定版本：当前型号下稳定可推荐版本，可按业务决定是否只允许一个

## 4. 业务流程

### 4.1 开发流程

```text
上传新包
  -> 待测试
  -> QA 点击开始测试
  -> 测试中
  -> 测试结果
     -> 通过 -> 测试通过
     -> 未通过 -> 测试未通过
```

补充说明：

- 测试未通过后，建议开发上传新包形成新的开发版本记录
- 如果业务上需要，也可以支持“重新提交测试”，把状态从测试未通过改回测试中
- 不建议直接覆盖原有失败包，失败包应保留在系统中，作为完整历史的一部分

### 4.1.1 测试未通过后的重新上传规则

测试未通过后，开发重新修复并上传新包时，建议遵循以下原则：

- 不覆盖原有固件包
- 新增一条新的开发版本记录
- 原失败版本继续保留，状态停留在“测试未通过”
- 新上传版本重新进入“待测试”

这样做的好处是：

- 能完整保留每一轮测试失败和修复的历史
- 能明确区分“失败包”和“修复后的新包”
- QA 和开发对版本记录的理解一致
- 后续追查问题时，能知道某个问题是在哪一轮修复的

示例：

- `0.0.25` -> 测试未通过
- `0.0.26` -> 修复后重新上传，进入待测试

建议日志中至少保留：

- 哪个版本测试未通过
- 失败原因
- 谁提交了新的修复包
- 新包对应的版本号和时间

### 4.2 提交发布流程

```text
测试通过
  -> 提交发布
  -> 待发布
  -> 管理员处理
     -> 发布
     -> 驳回到测试中
```

补充说明：

- 驳回只发生在“待发布”阶段
- 已发布后不允许再驳回

### 4.3 发布流程

```text
待发布
  -> 点击发布
  -> 进入发布版本
  -> 管理员选择发布类型
     -> 最新版本
     -> 稳定版本
```

系统规则建议：

- 如果发布类型为“最新版本”，同型号原来的最新版本自动取消最新标记
- 如果业务要求“稳定版本”只能有一个，则新标记稳定版本时自动取消旧稳定版本
- 如果业务允许多个稳定版本，则稳定版本只作为普通标记，不自动取消旧值

## 5. 状态与动作规则

### 5.1 开发版本允许的动作

- 待测试
  - 开始测试 -> 测试中
- 测试中
  - 测试通过 -> 测试通过
  - 测试未通过 -> 测试未通过
- 测试通过
  - 提交发布 -> 待发布
- 测试未通过
  - 上传新版本
  - 重新提交测试 -> 测试中
- 待发布
  - 发布 -> 已发布
  - 驳回 -> 测试中

### 5.2 发布版本允许的动作

- 已发布
  - 设为最新版本
  - 标记稳定版本
  - 作废
- 已作废
  - 仅允许查看历史

明确限制：

- 已发布版本不能删除
- 已发布版本不能驳回
- 已作废版本不能再次发布，除非业务明确要求恢复逻辑

## 6. 数据结构建议

不建议新建两张完全独立的“开发版本表”和“发布版本表”。

更合理的做法是：

- 继续保留一张固件版本主表
- 在主表上区分开发状态和发布状态
- 再增加发布历史表

### 6.1 固件版本主表建议新增字段

以当前 `alpha_firmware_versions` 为基础，建议增加或调整：

- `dev_status`
  - 开发状态
  - 值建议：`pending_test` / `testing` / `test_passed` / `test_failed` / `pending_release`
- `publish_status`
  - 发布状态
  - 值建议：`unpublished` / `published` / `voided`
- `is_latest`
  - 是否最新版本
- `is_stable`
  - 是否稳定版本
- `published_at`
  - 发布时间
- `published_by`
  - 发布人
- `voided_at`
  - 作废时间
- `voided_by`
  - 作废人
- `void_reason`
  - 作废原因
- `source_firmware_id`
  - 来源固件版本 ID，可选，用于表示“该版本是基于哪个失败版本修复后重新上传的”

说明：

- 现有 `status` 字段建议后续逐步迁移为 `dev_status + publish_status`
- 过渡期可以保留 `status`，但不建议长期继续单字段承载全部业务
- `source_firmware_id` 不是第一阶段必做字段，但如果希望把“失败后修复再上传”的链路表达得更清楚，建议保留

### 6.2 型号固件关系表建议保留

继续保留 `alpha_model_firmware_rels`，用于管理：

- 哪个型号关联了哪个固件版本
- 当前是否为该型号的当前发布版本

建议保留或强化字段：

- `model_id`
- `firmware_id`
- `is_recommended`

建议把前端文案统一成：

- `当前发布`

即：

- `is_recommended` 在业务上视为“当前发布版本”

### 6.3 发布历史表

建议新增：

- `alpha_firmware_release_records`

建议字段：

- `id`
- `firmware_id`
- `model_id`
- `release_type`
  - `latest` / `stable`
- `operator`
- `released_at`
- `remark`

这张表专门记录：

- 谁发布了
- 什么时候发布
- 发布成什么类型
- 对应哪个设备型号

### 6.4 状态流转日志表

现有日志表继续保留，用于记录：

- 上传固件
- 开始测试
- 测试通过
- 测试未通过
- 失败后重新上传新包
- 提交发布
- 驳回
- 发布
- 作废
- 设为最新版本
- 标记稳定版本

建议日志里保留：

- 原状态
- 目标状态
- 操作人
- 操作说明
- 对应型号

## 7. 页面设计建议

### 7.1 设备树页面

继续使用当前树型结构：

- 设备类别
  - 设备型号
    - 固件版本

这个结构不需要推翻，可以继续用。

### 7.2 固件节点展示建议

固件节点建议展示：

- 版本号
- 开发状态
- 发布状态
- 是否最新版本
- 是否稳定版本
- 是否当前发布

### 7.3 固件节点操作建议

根据版本当前所处阶段，动态显示操作按钮。

开发阶段常见按钮：

- 开始测试
- 测试通过
- 测试未通过
- 提交发布
- 驳回到测试中

发布阶段常见按钮：

- 发布
- 设为最新版本
- 标记稳定版本
- 设为当前发布
- 作废
- 查看日志

### 7.4 删除规则

建议界面规则：

- 未发布版本可以删除
- 已发布版本不显示删除按钮
- 已发布版本只显示“作废”

## 8. 推荐的字段映射

为了兼顾前后端实现，建议统一使用以下代码值。

### 8.1 开发状态

- `pending_test` -> 待测试
- `testing` -> 测试中
- `test_passed` -> 测试通过
- `test_failed` -> 测试未通过
- `pending_release` -> 待发布

### 8.2 发布状态

- `unpublished` -> 未发布
- `published` -> 已发布
- `voided` -> 已作废

### 8.3 发布类型

- `latest` -> 最新版本
- `stable` -> 稳定版本

## 9. 对当前系统的改动评估

本次不是推倒重来，属于在现有设备固件模块上的一次结构升级。

### 9.1 可直接复用

- 设备类别
- 设备型号
- 固件版本主表
- 型号固件关系表
- 固件日志
- MinIO 上传能力
- 当前树形页面基础结构

### 9.2 需要调整

- 固件状态设计
- 固件节点按钮逻辑
- 发布动作
- 发布历史记录
- 删除逻辑
- 文案与显示规则

### 9.3 改动级别判断

本次改动建议定义为：

- 中等改造

原因：

- 数据骨架还在，不是从零开始
- 但状态模型、流程动作、历史记录都要补

## 10. 推荐开发顺序

### 第一步：先改状态设计

- 新增 `dev_status`
- 新增 `publish_status`
- 保留兼容旧字段的迁移方案

### 第二步：补发布记录表

- 新建 `alpha_firmware_release_records`
- 落地发布人、发布时间、发布类型、对应型号

### 第三步：补后端动作接口

- 开始测试
- 测试通过
- 测试未通过
- 提交发布
- 驳回
- 发布
- 标记最新版本
- 标记稳定版本
- 作废

### 第四步：改前端树节点操作

- 根据状态动态显示按钮
- 已发布版本隐藏删除按钮
- 已发布版本显示作废按钮

### 第五步：补历史页或日志抽屉

- 状态流转日志
- 发布历史记录

## 11. 建议结论

最终建议方案如下：

- 保留当前“类别 -> 型号 -> 固件”树结构
- 把版本管理拆成“开发版本”和“发布版本”两条业务线
- 已发布后不允许驳回
- 已发布后不允许删除，只能作废
- 最新版本作为唯一标记
- 稳定版本作为手动标记
- 发布动作写入独立发布历史

这套方案和当前项目的现状兼容度较高，可以作为下一阶段代码改造的正式目标。
