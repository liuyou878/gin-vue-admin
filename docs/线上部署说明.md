# 线上部署说明

本文用于记录当前项目从本地打包到线上服务器部署的基本方式，方便后续交接和排查。

## 1. 部署原则

- 前端只部署打包后的静态文件
- 后端只部署可执行文件、生产配置和运行资源
- 线上和本地使用不同的数据库配置
- 前后端尽量分目录管理，避免互相覆盖

## 2. 推荐目录结构

建议线上服务器使用独立目录，例如：

```bash
/opt/alpha_erp/
  web/
    dist/
  server/
    server
    config.release.yaml
    resource/
```

如果你已经用了别的目录，也可以保持现状，只要结构清晰即可。

## 3. 前端打包

前端目录是 `web`，生产构建命令：

```bash
cd web
npm install
npm run build
```

构建完成后会生成 `web/dist/`，线上直接部署这个目录即可。

### 3.1 需要注意的点

- `web/openDocument.js` 只在开发启动时执行，不影响生产构建
- `web/vite.config.js` 里的 `Banner(...)` 只是构建注释，不影响页面功能
- `svgBuilder(...)` 会参与生产构建，相关图标资源和构建产物需要一起检查

## 4. 后端打包

后端目录是 `server`。如果要在 Linux 服务器上运行，建议在本地交叉编译：

```powershell
cd D:\afg\Alpha_ERP\server
$env:GOOS="linux"
$env:GOARCH="amd64"
$env:CGO_ENABLED="0"
go build -o server .
```

如果只是在 Windows 本机调试，可以直接：

```powershell
go build -o server.exe .
```

## 5. 生产配置

生产环境建议单独使用 `server/config.release.yaml`，不要直接复用开发配置。

生产配置重点检查这些内容：

- `mysql.path`
- `mysql.port`
- `mysql.db-name`
- `mysql.username`
- `mysql.password`
- `redis`
- `system.addr`

启动后端时优先使用生产配置，例如：

```bash
GIN_MODE=release ./server -c config.release.yaml
```

## 6. Nginx 反向代理

常见做法是：

- Nginx 托管前端 `dist`
- Nginx 将 `/api` 转发到后端

示例：

```nginx
location / {
    root /opt/alpha_erp/web/dist;
    try_files $uri $uri/ /index.html;
}

location /api {
    rewrite ^/api/(.*)$ /$1 break;
    proxy_pass http://127.0.0.1:8888;
}
```

如果后端不是本机运行，把 `proxy_pass` 改成实际后端地址即可。

## 7. 部署检查清单

上线前建议确认：

- 前端 `dist` 已重新打包
- 后端可执行文件是最新版本
- `config.release.yaml` 已切到线上数据库
- `resource/` 目录已一起同步
- Nginx 配置已指向新的静态目录和后端地址
- 浏览器缓存已清理，避免看到旧页面

## 8. 常见问题

### 8.1 为什么线上和本地数据库不一样

因为生产环境应该单独使用线上库配置，开发环境继续保留本地库配置，两个环境不要混用。

### 8.2 为什么打包后和开发环境看到的页面不一样

生产构建会经过 Vite 插件和压缩流程，某些构建阶段逻辑只会在打包后生效，开发环境不一定完全一致。

### 8.3 为什么页面资源或图标缺失

先检查：

- `web/dist` 是否完整
- `web/src/assets/icons` 是否随构建被正确打包
- `svgIcon` 和相关插件是否还在使用
